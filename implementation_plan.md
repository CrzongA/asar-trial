# Autonomous Search and Rescue (ASAR) Simulation Framework

Provide a technical outline and implementation plan for an ASAR framework featuring a web frontend, ROS for communication, a high visual fidelity simulator, and a locally-hosted agentic Vision Language Model (VLM).

> [!NOTE]
> **Deployment Target**
> This architecture is specifically designed for deployment on an **AMD MI300X** accelerator. This hardware profile provides massive compute and VRAM (~192GB), completely eliminating standard resource constraints, but restricts the use of software hard-locked to NVIDIA's CUDA/RTX ecosystems.

> [!CAUTION]
> **Latency vs Real-Time Control & The Reason-Act-Observe Loop**
> VLMs inherently possess high inference latency. A drone requires high-frequency control loops (50Hz+). The VLM *cannot* be used for low-level flight control or immediate obstacle avoidance. 
> 
> **Solution:** Implement a **Reason-Act-Observe** workflow loop. 
> - **Observe:** The camera node periodically captures frames (e.g., 1-2 FPS) and sends them to the VLM.
> - **Reason:** The VLM analyzes the frame asynchronously (e.g., semantic matching of textually-described targets, translating natural language commands into mission tasks) and updates a semantic map.
> - **Act:** A dedicated high-frequency flight controller (**PX4**) handles the actual flying and obstacle avoidance, guided by high-level waypoints generated by the "Reason" phase.

## Proposed Architecture

### 1. Simulator Environment: Gazebo (Harmonic) + PX4 Autopilot
- **Why**: Since the target hardware is an AMD MI300X, **NVIDIA Isaac Sim cannot be used** as it relies heavily on proprietary NVIDIA RTX and CUDA technologies. Gazebo (Harmonic version) utilizes the Ogre2 rendering engine which has excellent support for standard graphics APIs (Vulkan) and runs exceptionally well on AMD GPUs. Modern Gazebo offers strong physics and vastly improved visual fidelity compared to its predecessors.
- **Flight Stack Integration**: The simulator will use **PX4 Software-In-The-Loop (SITL)** to provide realistic flight dynamics and control interfaces.

### 2. Middleware: ROS 2 (Jazzy)
- **Why**: ROS 2 Jazzy is the latest LTS release, offering the best performance, security, and long-term support for modern robotics.
- **Usage**: Will handle passing telemetry, video streams, VLM inference results, and mission planner commands between the frontend, the simulator, and the AI agent via `uxrce_dds` (Micro-XRCE-DDS) for PX4 communication.

### 3. Agentic Vision Language Model (VLM): Qwen3-VL via vLLM
- **Suggested Model**: Qwen3-VL-235B-A22B-Instruct-FP8
- **Why Qwen3-VL**: Offers state-of-the-art vision-language reasoning capabilities.
- **Why vLLM**: vLLM provides a massive performance boost over standard `llama.cpp` deployments by implementing **PagedAttention** and continuous batching. On a data-center class GPU like the MI300X, vLLM fully saturates the massive memory bandwidth, drastically increasing token throughput and reducing latency. It provides official "day-0" support for ROCm (AMD's compute stack).
- **Core Tasks**:
  1. **Semantic Matching**: Locating textually-described targets (e.g., "person wearing a red jacket", "damaged white vehicle") within the drone's camera feed.
  2. **Task Translation**: Translating high-level natural language commands (e.g., "Search the northern perimeter for survivors") into executable mission tasks and waypoint paths.
- **Integration**: We will deploy Qwen3-VL using the official ROCm vLLM Docker container. A dedicated ROS 2 Jazzy node will capture images, query the local vLLM REST API, and publish the semantic bounding boxes and translated mission parameters.

### 4. Frontend: Next.js + roslibjs + WebRTC
- **Why**: A React-based Next.js application provides a robust structure for a complex dashboard.
- **Visuals**: Will feature a premium, dynamic UI (dark mode, glassmorphism, responsive grid).
- **Communication**: Use `roslibjs` communicating over `rosbridge_server` (WebSockets).
- **Video Feed**: Use WebRTC for low-latency, high-quality video streaming to the browser.
- **Mission Planner**: Integrate Leaflet or Mapbox to visualize the drone's GPS coordinates, mission waypoints, and markers generated by the VLM.

## Proposed Changes

### ROS Workspace
#### [NEW] `ros_ws/src/vlm_agent/`
A ROS 2 package executing the **Observe-Reason** logic. It captures simulator camera frames, queries the Qwen3-VL REST API, executes semantic matching, and parses natural language commands into mission tasks.
#### [NEW] `ros_ws/src/mission_control/`
A ROS 2 package containing the **Act** logic state machine. It consumes translated VLM tasks, manages the mission state, and handles offboard control waypoint generation for PX4.

### Frontend Setup
#### [NEW] `frontend/`
A Next.js application initialized using `create-next-app`. Will contain components for:
- `VideoPlayer.jsx`: WebRTC stream of the drone's POV.
- `TelemetryDashboard.jsx`: Altitude, speed, battery, etc.
- `MissionMap.jsx`: 2D map with path and markers.
- `VLMConsole.jsx`: Live feed of the VLM's interpretation of the scene and reasoning logs.

### Simulator Configuration
#### [NEW] `sim_config/`
Launch files and scripts to bootstrap the Gazebo Harmonic environment, spawn the drone, instantiate PX4 SITL, and attach the ROS 2 Jazzy camera and IMU sensors.

### Infrastructure & Orchestration
#### [NEW] `infrastructure/`
Contains necessary middleware configurations and orchestration scripts:
- **`docker-compose.yml`**: Orchestrates the ROCm vLLM Docker container and defines network boundaries for the AI services.
- **ROS-Web Bridge**: Launch files for `rosbridge_server` to establish the WebSocket connection required by the Next.js `roslibjs` client.
- **WebRTC Streamer**: Configuration or node setup (e.g., via GStreamer/WebRTC bridge) to capture the Gazebo ROS image topic and stream it to the frontend via WebRTC.
- **Micro-XRCE-DDS Agent**: Startup scripts for the DDS agent, which is mandatory for PX4 SITL to communicate with the ROS 2 Jazzy network.

## Verification Plan

### Automated Tests
- Write ROS 2 unit tests to verify that the `vlm_agent` correctly parses mock Qwen3-VL API responses (target matching & command translation) and publishes the correct custom ROS messages.

### Manual Verification
- **Module Bringup**: Start ROS 2, vLLM (ROCm Docker), PX4 SITL, and Gazebo. Verify via `rqt_image_view` that the camera feed is active.
- **Reason-Act-Observe Pipeline**: Fly the drone manually in Gazebo. Send a natural language command to the VLM via a ROS topic. Verify the system generates correct waypoints. Verify that semantic targets are correctly bounding-boxed.
- **Frontend Integration**: Open the Next.js app, verify the video feed plays smoothly, telemetry updates in real-time, and the VLM console populates with the agent's insights.
